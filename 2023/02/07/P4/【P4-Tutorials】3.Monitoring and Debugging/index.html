3.1 Explicit Congestion Notification（显式拥塞控制）

通过显式拥塞通知（ECN）的实现扩展基本的三级转发。ECN 允许端到端通知网络拥塞而不丢弃数据包。如果终端主机支持 ECN，它会在 ipv4.ECN 字段中输入值 1 或 2。对于这样的分组，如果队列大小大于阈值，则每个交换机可以将值更改为 3。接收方将值复制给发送方，发送方可以降低速率。

网络拓扑图：

h1 到 h2 发送低速流量，h11 到 h22 通过 iperf 发送高速流量。并通过 topology.json 文件限制 s1 和 s2 之间的链路 512kbps。这时在 h2 抓包观察 ECN 的置位情况。

首先需要对 ip 头进行处理，将 8bit 的 tos 位变成 6bit 的 diffserv 和 2bit 的 ecn。

然后判断队列长度，如果长度大于阈值，那么将 ecn 位置为 3。

然后要记得更新校验和的计算内容。

然后进行测试:

  • 首先使用命令行 xterm h1 h11 h2 h22 进入各个主机命令行界面
  • h2 运行命令 ./receive.py > h2.log 接收数据包，并且将输出写入 h2.log 中
  • h22 运行命令 iperf -s -u 作为服务端进行 udp 测试
  • h1 运行命令 ./send.py 10.0.2.2 "P4 is cool" 30 给 h2 发送 30 秒的数据包
  • 同时 h11 运行命令 iperf -c 10.0.2.22 -t 15 -u 作为客户端向 10.0.2.22 即 h22 传输 15 秒的数据
  • 最后使用命令 cat h2.log |grep tos 查看 h2 收到的数据包中 ecn 位的情况

可见其中发生了拥塞，ECN 位被置成 3 了。

3.2 Multi-Hop Route Inspection（多跳路由检测）

用带内网络遥测（INT）的缩小版本扩展基本的三级转发，称之为多跳路由检测（MRI）。
MRI 允许用户跟踪每个数据包经过的路径和队列长度。为了支持此功能，需要将交换机的 ID 和队列长度附加到每个数据包的头堆栈。在目的地，交换机 ID 的序列对应于路径，每个 ID 后面都是交换机端口的队列长度。

网络拓扑图和 3.1 一致，而且 s1 到 s2 的带宽限制也一致：

直接使用 3.1 的测试命令，在 h2 得到接收的数据包信息：

可见其中没有显示任何跳信息。

根据 mri.p4 中头的定义，MRI 将自定义两个 header。 第一个头部 mri_t 包含单个字段 count，该字段经过的路由器的数量。 第二个头部 switch_t 包含数据包经过的每个交换机跳的交换机 ID 和队列深度字段。实现 MRI 的最大挑战之一是处理用于解析这两个标头的递归逻辑。将使用 parser_metadata 的字段 remaining 来跟踪需要解析多少个 switch_t 标头。在 parse_mri 状态下，此字段应设置为 hdr.mri.count。其中头部的 swtraces 保存经过的路由器序列，每个序列保持路由器的 ID 和队列深度。

首先是 Paser 部分，首先确认是否需要解析 mri 头，如果需要解析 mri 头，其中 count 字段表示路由器序列的数量，因此需要循环解析，每解析一个 switch_t 的 id 和队列深度，就将 remaining 字段减 1，继续循环直到所有的 switch_t 都被解析完。

然后是 Egress 部分，对于经过此路由器的数据包，需要将此路由器的 ID 和深度也保存到数据包的 swtraces 中，而且 mri 头需要将 count 字段加 1。

重新测试，可以发现，s1 的队列长度居然有 38，由于 s1 到 s2 的带宽限制，s1 出现了拥塞，而通过 mri 实现了实时监控数据流途径节点以及节点的拥塞情况。
