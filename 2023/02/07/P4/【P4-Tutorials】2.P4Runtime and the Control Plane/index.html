2.1 P4Runtime（控制器）

1）本次实验基于 1.2 实现，但是增加两个 counters(ingressTunnelCounter, egressTunnelCounter) 和两个新的 actions(myTunnel_ingress, myTunnel_egress)

2）关于控制平面，P4 程序只是定义包处理的 pipeline，而 table 的 rules 是需要控制平面下发。在本次实验里，mycontroller.py 将实现控制平面，这点和前两个实验有所不同

也就是说原来的路由器的表项是由 sN-runtime.json 这个文件规定的，现在通过 mycontroller.py 这个文件来设置。

网络拓扑图如下所示：

首先不修改代码运行，并进行 ping 测试：

不成功！在 ping 的同时开启另一个终端，运行 mycontroller.py，这将在交换机上安装程序，并推送隧道 ingress 的规则，这个程序每两秒打印隧道 ingress 和 egress 计数器的值，可以看到 s1 交换机的 ingress tunnel 计数器呈增长状态。也就是说 s1 收到到数据包，但是没有转发出去，因此 s1 的 egress 和 s2 的两个计数器的值都是 0，没有变化。

每一个交换机都根据目标地址 ip 地址将流量映射到 tunnel 中，我们的任务是根据 tunnel ID 编写交换机之间的流量。

mycontroller 相当于控制平面，对交换机进行配置，其主要功能为：

  • 为 P4Runtime 服务建立到交换机的 gRPC 连接
  • 将 P4 程序推到每个交换机
  • 为 h1 和 h2 之间的两个隧道编写隧道入口和隧道出口规则，也就是 s1 和 s2 两个路由器的 rules
  • 每两秒读取隧道入口和出口计数器

在这个实验过程中，可能用到一些 dictionary 中的类和方法：

  • helper.py：包含用于解析 p4info 文件的 P4InfoHelper 类，提供实体名称与 ID 的转换方法，构建 P4Runtime table 的 P4 程序相关部分；
  • switch.py：包含一些获取 gRPC 客户端的存根，并建立到交换机的连接，提供 helper 方法，用于构造 P4Runtime 协议缓冲区消息并进行 P4Runtime gRPC 服务调用；
  • bmv2.py：扩展了 SwitchConnections，并提供 BMv2 特定的设备载荷以加载 P4 程序；
  • convert.py：提供便捷的方法，将字符串和数字编码、解码成协议缓冲区消息所需的字节字符串。

首先看一下 advance_tunnel.p4：

整个过程就是路由器收到数据包之后，如果没有隧道头，也就是说 myTunnel 是 InValid 的，那么根据数据包的目的地址在表 ipv4_lpm 中匹配，匹配到的动作为 myTunnel_ingress，也就是执行隧道入口的动作，设置 myTunnel 头，根据目的 ip 地址，赋予 dst_id，比如说 h1 发一个数据包，目的地址是 h3，这个时候 s1 收到了，根据 h3 的目的地址，执行隧道入口动作，并且 action params 是 h3 的 dst_id=3；然后设置 myTunnel 头为 valid，然后查表 myTunnel_exact，根据 myTunnel 头里的 dst_id 匹配，如果匹配到的动作是 myTunnel_forward，也就是转发到下一跳的路由器，就直接根据下一个路由器的端口进行转发，如果匹配到的动作是转发到主机，也就是动作 myTunnel_egress，并得到主机的地址和端口，然后设置 myTunnel 头为 InValid，修改以太头，进行转发。回到刚刚的例子，由于数据包隧道头已经是 Valid 的了，进入 myTunnel_exact 查询，根据 dst_id=3 执行隧道转发的动作，并且直接从端口 3 发出去，然后 s2 收到数据包，由于隧道头是 Valid 的，进入 myTunnel_exact 查询，根据 dst_id=3 执行隧道出口的规则，得到主机 h3 的 MAC 地址和端口，转发给主机 h3。整个过程每执行隧道入口和隧道出口一次，计数器就会记录包的大小和个数。

然后在 mycontroller.py 中补充函数 writeTunnelRules，即实现给路由器设置 rules，主要需要编写的是 tunnel transit rule：匹配隧道 id 并将数据包转发到下一跳路由器。
