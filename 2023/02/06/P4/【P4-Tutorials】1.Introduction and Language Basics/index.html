1.1 Basic Forwarding （基础 ipv4 转发）

1）实现基本 ipv4 转发

2）交换机必须对每个数据包执行以下操作:

  • 更新源和目标 MAC 地址
  • 减少 IP 报头中的生存时间（TTL）
  • 将数据包转发到适当的端口

网络拓扑图根据 topology.json 定义，如下图：

需要实现的部分为解析 Parser、Ingress Processing、逆解析 Deparser 三部分。

Parser 部分：主要要解析以太网头和 ip 头，数据包转发不涉及 tcp 层，所以可以不解析。

Ingress processing 部分：根据解析得到的 ip 头的目的 ip 地址来匹配动作，简单来说就是对目的 ip 地址进行最长前缀匹配，实现查表转发，如果匹配到的动作是 ipv4_forward，也就是转发，而且动作的参数是 MAC 地址和端口，那么这个动作就是把这个数据包以太头的源 MAC 地址变成以太头的目的 MAC 地址，也就是这个交换机的 MAC 地址，把目的 MAC 地址设置为参数中的 MAC 地址，也就是这个交换机根据目的 ip 地址匹配到的动作参数之一，其实就是下一跳的 MAC 地址，端口同理；然后设置 TTL - 1。

Deparser 部分：这部分就是重新打包数据包，按照拆包的顺序，parser 先解析哪个包头，就先打包哪个包头。

1.2 Basic Tunneling（隧道传输）

在上面 ipv4 转发的基础上，添加基本隧道传输功能。

1）定义一个新的报头类型 myTunnel_t 来封装 ip 数据包并修改交换机代码，以便它使用新的隧道报头来决定目标端口

2）新的头类型将包含两个 16 位字段 proto_id 和 dst_id。myTunnel 报头对应的 etherType 为 0x1212，该 proto_id 指示要封装的数据包的类型，以及用于路由的目的地 id

3）定义一个名为 myTunnel_forward 的新动作，它简单地将出口端口（即 standard_metadata 总线的 egress_spec 字段）设置为控制平面提供的端口号

4）如果 myTunnel 报头是有效的，更新 MyIngress 控制块中的 apply 语句来应用你新定义的 myTunnel_exact 表；否则，如果 ipv4 报头有效，则调用 ipv4_lpm 表

网络拓扑图如下所示，每台主机对应一个隧道 id，即 dst_id。

查看 s1-runtime.json，可以发现，myTunnel_exact 这个表的作用就是，按照隧道头中的目的隧道 dst_id 字段来进行转发，不管 ip 地址是什么。比如 h1 发送数据包，设置隧道头中的 dst_id 字段为 3，则 s1 不管这个数据包的 ip 地址是多少，根据 dst_id=3 匹配到的端口值是 3，那么直接从端口 3 转发出去。

需要实现的部分为解析 Parser、Ingress Processing、逆解析 Deparser 三部分。

Parser 部分：增加一个头类型，需要先看看是不是 myTunnel 并解析，然后再解析 ip 头。

Ingress processing 部分：要看看隧道头有没有失效，如果失效了，则进行 ipv4 的查表转发，如果没失效，就根据隧道头的 dst_id 查表得到转发的端口。
