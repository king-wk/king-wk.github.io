0.2 以 basic.p4 为例解读 p4 文件结构

这是 P4 中提供的最简单最易理解的编程结构，V1Model。可以看到它由 5 个模块组成：

  • Parser：解析器， 解析并且提取数据包头的各个字段，看看包里有啥，解包的层数也可以自己定义（mac->ip->tcp…）
  • Ingress：Ingress 处理，在这里定义 Ingress 流水线，看看哪些 bits 组成的 fields 有没有我们想要的，比如目标 IP 是不是 8.8.8.8，是的话就改写一下 header 让包能被转发到它该去的地方
  • TM：Traffic manager，有一些队列，用于流量控制（一些队列相关的 metadata 在此更新）
  • Egress： Egress, 在这里定义 Egress 流水线，比如改写源 MAC 地址，同时提供更多对网络包的操作空间
  • Deparser：用于重组数据包，因为数据包在处理过程中经历了分解和处理，所以最后转发的时候需要重组一下

P4 为我们提供了上述抽象，我们就可以把所有的交换机理解为上述的模型，然后按照上述模型进行开发就可以了。

下面以 basic.p4 为例理解每一部分干了啥，basic.p4 主要包括：头部定义、解析 Parser、校验和检验、入口处理 Ingress processing、出口处理 Egress processing、检验和计算、逆解析 Deparser。

parser 解析数据包：parser 是一个有限状态机。从 start 状态开始，每一个状态便解析一种协议，然后根据低层协议的类型字段，选择解析高一层协议的状态，然后 transition 到该状态解析上层协议，从 MAC 头到 IP 头到 TCP 头等等，最后 transition 到 accept 或者 reject；

Ingress：Ingress 中，要实现一个转发功能，因此需要定义一个用于转发的流表；basic.p4 中定义的流表可以看作一个如下的表，就是以数据包的目的 ip 地址为 key 来匹配表中的表项，看对应的动作是什么，比如动作是 ipv4_forward，对应的 action_params 是下一跳的 mac 地址和端口；而 Egress 同样可以定义这样的表，可以定义多个表，进行多次处理。

key action action_params
“10.0.1.1”, 32 ipv4_forward 08:00:00:00:01:11, 1
… … …

Checksum 和 Deparser：这两个部分都有高度抽象的内置函数直接完成；

控制面代码在 pod-topo 目录下面 s1-runtime.json、s2-runtime.json、s3-runtime.json、s4-runtime.json，定义好了每个交换机的流表项，即定义了对应交换机具体的配置方案，匹配哪些域进行哪些动作，这些动作的集合（比如 ipv4_forward、drop 等等）在 basic.p4 文件中进行了定义。

下面是 s1-runtime.json，可见其定义了每一个表的默认动作，表的每一条表项，match 的内容是什么，对应什么动作，动作的参数等等。
